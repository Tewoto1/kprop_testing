# Cumulant propagation vs Monte-Carlo — testing notebook

This folder tests the **real** cumulant-propagation ("kprop") algorithm in
`src/mlp_kprop` (treated as a black box) against Monte-Carlo, for simple MLPs.
The distribution is over the **input** `X ~ N(0, I_input_dim)`; the model weights
are **fixed** after training.

## Start here: the Colab notebook

**`notebooks/cumulant_experiments_colab.ipynb`** is the primary, self-documenting
way to run everything. It states every assumption and variable and gives a
verification cell for each. Sections:

- §0–1  Setup (get repo, minimal deps, float64/seed/device)
- §2    **Verify the real algorithm** — prints the source file of `mlp_kprop`
- §3    All assumptions + one editable cell with **every variable**
- §4    Adapter walkthrough with `debug=True` (extracted/converted shapes)
- §5    Sanity checks
- §6    One model end-to-end (kprop mean vs MC mean, metrics)
- §7    **Q1** — does kprop predict the mean after training to zero?
- §8    Optional: run the full Q1 CLI sweep and show saved plots
- §9    **Q2 & Q3** — width-scaling (initial vs trained, the `k_max` ladder).
        Here **`input_dim = width`** (the input dimension scales with the hidden
        width), so *every* layer benefits from increasing `n` and the theoretical
        `n^(-k_max/2)` slope is not bottlenecked by a fixed input dimension.
- §10   Cautious conclusions

To run on Colab: push this repo to GitHub, set `REPO_URL` in §0 (or mount Drive
and set `LOCAL_REPO_DIR`), then Run all. Sweeps default to reduced sizes;
comments say which variables to bump for the full run.

The notebook is generated by `notebooks/build_notebook.py` (edit there + rerun to
regenerate, so the notebook stays reproducible).

## The three questions it answers

- **Q1 — Mean after training to zero.** Does kprop still predict `E_X[f(X)]`?
- **Q2 — Width scaling at init.** Does error fall like `n^(-k_max/2)`, and does
  raising `k_max` buy accuracy?
- **Q3 — Does training break the scaling / `k_max` refinement?**

## Library (what the notebook imports)

- `cumulant_adapter.py` — `run_cumulant_propagation_from_model(model, input_dim,
  cumulant_config, device)`: extracts the MLP's `(weight, bias)` per layer,
  **verifies orientation** (PyTorch `nn.Linear` is `(out, in)`; kprop contracts
  over `in`, so **no transpose**), builds `K_in = {1: zeros(n), 2: eye(n)}` for
  `X ~ N(0, I)`, calls the real `mlp_kprop`, and returns
  `{"raw_output", "mean", "metadata"}`. Also holds the sanity checks.
- `model_utils.py` — `make_mlp` (repo `MLP`), `train_model_to_zero`
  (MSE against the all-zeros target on fresh Gaussian inputs), seeding, norms.
- `metrics.py` — `estimate_empirical_mean` (batched float64 MC) and
  `compare_means`.
- `plotting.py` — width-scaling plots (used by the CLI runner below).

The model is the repo's own `src.mlp_kprop.mlp.MLP`, because `mlp_kprop`
consumes an `MLP` instance — so the trained model and the model kprop sees are
identical.

## Command-line runner (used by notebook §8)

`experiments/run_cumulant_train_to_zero_mean.py` runs the full Q1 sweep, writing
an incremental CSV, `config.json`, and plots:

```bash
uv run python experiments/run_cumulant_train_to_zero_mean.py \
    --input-dim 64 --output-dim 1 --hidden-depth 3 \
    --widths 64 128 256 512 --seeds 0 1 2 3 4 \
    --mc-samples 64000 --train-steps 5000 --k-max 3 \
    --outdir results/cumulant_train_to_zero
```

Flags: `--skip-training`, `--no-plots`, `--no-sanity`, `--debug`, `--no-bias`,
`--use-avg-metric`, `--reanalyze` (recompute metric columns + plots from an
existing CSV with no model runs).

## Key assumptions & numerics

- **float64** everywhere (kprop is run in double precision in the repo's tests).
- **`k_max`** (budget `K`) is a free knob — **no hard cap**. Higher = more
  accurate but ~`O(n^k_max)` memory/time, so very large `k_max` can OOM at large
  width on a small machine (fine on a Colab GPU). `factor=True` is only used for
  `k_max ∈ {3,4}` and is auto-disabled otherwise. `output_d_max=1` (mean only).
  `use_avg_metric=False` — the exact metric `W·metric·Wᵀ` from the **fixed**
  weights, not the init-time `E[WWᵀ]`.
- Activation must be a key of `WICK_COEF_D` (relu/gelu/tanh/sigmoid/square/cube/
  heaviside/sgn); default `relu`. Never silently swapped.

## Reading the metrics (important)

The trained mean is tiny, so normalization matters:

- **Headline:** `relative_error_mean = |cp − mc| / |mc| = √NMSE` (scale-free).
- **`mc_noise_z` = `|cp − mc| / MC_stderr`** — `z ≲ 1` means the gap is within MC
  sampling noise (kprop is at least as good as the MC reference); `z ≫ 1` means a
  real kprop bias.
- For width scaling, error is averaged over a **vector output** and **debiased**
  by subtracting the MC-variance floor (`mean stderr²`), so kprop error can be
  resolved *below* the sampling floor.
- Do **not** normalize by `E‖Y‖²` (or read raw NMSE alone): that denominator also
  collapses as the output → 0 and can make a *degrading* estimate look like it is
  improving.

## What the runs showed (cautious)

- **Q1:** at init the kprop mean is within MC noise (`z ≲ 1`); after training to
  zero a real bias appears (`z` up to ~6 at width 64). The smaller *absolute*
  error after training is a scale illusion (output is ~100× smaller); scale-free,
  training makes kprop **less** accurate.
- **Q2:** at init, error falls with width and the slope steepens with `k_max`
  (k=2→3 buys 6–15×). With input_dim *fixed* at 64 the measured exponents
  (~−0.5, −1.0) were shallower than the asymptotic `−k_max/2`; the §9 notebook now
  scales `input_dim = width` to remove that bottleneck and check whether the slope
  approaches the theoretical `−k_max/2`.
- **Q3:** training **disables the `k_max` refinement** — trained, k=2→3 buys ~1×
  (vs 6–15× at init) — consistent with training-induced correlations violating the
  wide-random-MLP assumptions behind cumulant propagation.
